# Kwork → Portfolio

Скилл для превращения сырых данных заказов Kwork (переписка, файлы, отзывы) в готовые
JSON-файлы портфолио, SVG-обложки и архитектурные схемы.

## Когда использовать

Пользователь даёт id заказа Kwork (или несколько id). Данные заказа уже собраны
в папке `kwork_projects/{id}/` и содержат:
- `{id}_raw.md` — переписка, системные сообщения, отзывы
- Прикреплённые файлы проекта (исходники, документы, скриншоты) — если есть

**Важно:** если в папке нет файлов кроме `{id}_raw.md`, значит срок хранения файлов
на Kwork истёк и скачать их невозможно. В этом случае анализируй проект только
по переписке: упоминания технологий, описания задач, фрагменты кода в сообщениях,
названия файлов. Не проси пользователя найти файлы — их нет.

## Входные данные

Пользователь указывает:
- **ID заказа** (одно число или список) — соответствует папке в `kwork_projects/`
- **Путь к папке kwork_projects** (если отличается от дефолтного)

## Алгоритм работы

### Шаг 1: Прочитай данные заказа

1. Открой `kwork_projects/{id}/{id}_raw.md`
2. Проверь, есть ли другие файлы в папке `kwork_projects/{id}/` (исходный код,
   документы, скриншоты, конфиги). Если папка содержит только `{id}_raw.md` —
   файлы проекта недоступны (срок хранения на Kwork истёк), анализируй только
   переписку
3. Составь полную картину проекта

### Шаг 2: Проанализируй и извлеки информацию

Из переписки и файлов определи:

**Основное:**
- Что за проект (бот, API, сервис, интеграция, парсер, автоматизация и т.д.)
- Какую задачу решал заказчик (его боль / проблема)
- Что конкретно было сделано (техническое решение)
- Какой результат получен (метрики, улучшения, автоматизация)

**Технический стек:**
- Языки программирования (определи из кода и переписки)
- Фреймворки и библиотеки
- Базы данных
- Инфраструктура (Docker, серверы, хостинг)
- Внешние API и интеграции

**Метаданные:**
- Отрасль заказчика
- Тип проекта (для тегов)
- Есть ли NDA-ограничения (если в переписке упоминается конфиденциальность
  или заказчик просил не раскрывать данные — ставь `isNda: true`)
- Отзыв заказчика

**Архитектура:**
- Компоненты системы
- Как они взаимодействуют
- Внешние сервисы и API
- Хранилища данных

### Шаг 3: Сгенерируй JSON

Создай файл по формату портфолио. Прочитай точную JSON-схему из
`references/project-schema.md` перед генерацией.

Ключевые правила для JSON:
- `slug` — английское короткое имя через дефис (например `hr-bot`, `payment-api`)
- `kwork_id` — числовой ID заказа Kwork (то число, из которого взята папка `kwork_projects/{id}/`). Ставь первым полем после `slug`
- `title` — если NDA, обезличи: отрасль + тип задачи (например «Бот автоматизации
  для логистической компании»)
- `brief` — 1-2 строки, понятные не-программисту. Формулируй на языке бизнес-ценности,
  а не технологий
- `case.problem` — опиши боль заказчика, почему он обратился. Не технический язык,
  а бизнес-проблема
- `case.solution` — что было сделано технически и ПОЧЕМУ было выбрано именно такое
  решение. Здесь можно и нужно быть техничным
- `case.result` — конкретные метрики и цифры. Если точных цифр нет — оцени на основе
  контекста переписки (сколько экономит времени, какой объём обрабатывает)
- `metrics` — 2-4 ключевые цифры. Если в переписке нет явных метрик, выведи их
  из контекста (количество пользователей, объём данных, скорость работы)
- `tags` — заполни все три категории: type, stack, industry
- `media.cover` — поставь заглушку `/assets/projects/{slug}/cover.svg`

### Шаг 4: Создай SVG-обложку

Создай файл `public/assets/projects/{slug}/cover.svg` размером **800×600**.

**Цветовая палитра по типу проекта:**

| Тип | Градиент (от → до) | Акцент |
|-----|-------------------|--------|
| бот / Telegram | `#0f2744 → #051020` | `#22d3ee` |
| API / сервис | `#0a2a2a → #041515` | `#0d9488` |
| e-commerce / магазин | `#1e0f44 → #0c0720` | `#7c3aed` |
| аналитика / SaaS | `#1a1000 → #0c0900` | `#f59e0b` |
| финтех / платёж | `#1a0808 → #0c0404` | `#ef4444` |
| CRM / интеграция | `#0a1a0a → #040c04` | `#10b981` |
| мониторинг / DevOps | `#0a0a20 → #04040f` | `#6366f1` |

**Структура SVG:**

```svg
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 800 600">
  <defs>
    <linearGradient id="bg" x1="0%" y1="0%" x2="100%" y2="100%">
      <stop offset="0%" stop-color="{GRAD_FROM}"/>
      <stop offset="100%" stop-color="{GRAD_TO}"/>
    </linearGradient>
  </defs>

  <!-- Фон -->
  <rect width="800" height="600" fill="url(#bg)"/>

  <!-- Сетка-оверлей (opacity 0.04–0.06) -->
  <g stroke="{ACCENT}" stroke-width="0.5" opacity="0.05">
    <!-- горизонтальные линии через каждые 60px -->
    <!-- вертикальные линии через каждые 80px -->
  </g>

  <!-- Тематические визуальные элементы (см. ниже) -->

  <!-- Частицы/точки (5–8 кругов, opacity 0.15–0.4, размер 3–8px) -->

  <!-- Декоративный акцент-элемент (линия, круг, угловой элемент) -->
</svg>
```

**Тематические визуальные элементы по типу:**

- **бот / Telegram**: чат-пузыри (roundrect) с горизонтальными полосками-текстом +
  иконка-аватар бота (circle + rect-«лицо» с dot-глазами)
- **API / сервис**: code-блок (rect с тёмным фоном) с endpoint-строками `/api/v1/...` +
  route-path (линии с узлами-кругами между блоками)
- **e-commerce / магазин**: phone-mockup (rect с rounded) с карточками товаров внутри +
  кнопка «Оплатить»
- **аналитика / SaaS**: metric-cards (3–4 карточки с числами и лейблами) +
  bar chart или line chart из `<rect>` / `<polyline>`
- **финтех / платёж**: card-mockup (кредитная карточка с chip и номером) +
  counter транзакций (число + подпись)
- **CRM / интеграция**: boxes двух-трёх систем с двусторонними стрелками между ними +
  лейблы интеграций
- **мониторинг / DevOps**: terminal-mockup (тёмный rect с заголовком и status-list) +
  uptime bar (зелёный прогресс-бар)

### Шаг 5: Создай архитектурную SVG-схему

Создай файл `public/assets/projects/{slug}/architecture.svg` с диаграммой компонентов.

**Шаблон SVG:**

```svg
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 900 420"
     style="background:#0d0d1a; font-family: 'Inter', sans-serif;">
  <defs>
    <marker id="arr" markerWidth="10" markerHeight="7" refX="9" refY="3.5" orient="auto">
      <polygon points="0 0, 10 3.5, 0 7" fill="{ACCENT}" opacity="0.7"/>
    </marker>
  </defs>

  <!-- Блоки: fill="#0f1f30" stroke="{ACCENT}" stroke-width="1.5–2" rx="8" -->
  <!-- Текст-название: fill="#e2e8f0" font-size="13" text-anchor="middle" -->
  <!-- Текст-подпись: fill="#94a3b8" font-size="10" text-anchor="middle" -->
  <!-- Технология/версия: fill="#818cf8" или fill="{ACCENT}" font-size="10" -->

  <!-- Основные стрелки: stroke="{ACCENT}" stroke-width="1.5" marker-end="url(#arr)" opacity="0.7" -->
  <!-- Второстепенные: добавь stroke-dasharray="5,3" opacity="0.5" -->

  <!-- Очереди/кэш: stroke="#f59e0b" -->
  <!-- Уведомления/алерты: stroke="#ef4444" или stroke="#10b981" -->
</svg>
```

**Правила расположения:**
- Внешние источники (пользователи, внешние API) — слева или сверху
- Основной сервис/бот — по центру, более толстая рамка (`stroke-width: 2`)
- БД и хранилища — справа от основного сервиса
- Уведомления/алерты — отдельная ветка сверху или снизу
- Dashboard/UI — крайний правый блок
- Максимум 6–8 блоков, не перегружай

**Подписи на русском или английском** (технические термины — на английском).

### Шаг 6: Сохрани результаты

Создай следующие файлы (используй Write tool для каждого):
1. `content/projects/{slug}.json` — основной JSON портфолио
2. `public/assets/projects/{slug}/cover.svg` — SVG-обложка 800×600
3. `public/assets/projects/{slug}/architecture.svg` — SVG-схема архитектуры 900×420

### Шаг 7: Покажи результат пользователю

Выведи:
1. Краткую сводку: что за проект, какой стек определён
2. JSON-файл целиком (для проверки)
3. Список созданных файлов
4. Вопросы, если что-то осталось неясным (NDA статус, точные метрики и т.д.)

Спроси пользователя: «Всё верно? Поправить что-нибудь?» перед тем как считать
проект завершённым.

## Обработка нескольких проектов

Если пользователь передал список id — обрабатывай по одному, показывая результат
каждого перед переходом к следующему.

## Частые ситуации

**Мало информации в переписке** — если переписка короткая (например «сделайте бота»
→ «готово») и файлы проекта недоступны (истёк срок хранения), извлекай максимум
из косвенных данных: названия файлов в сообщениях, упоминания технологий, описания
задач заказчиком. Если информации совсем мало — честно отметь это и предложи
пользователю дополнить вручную.

**Нет явных метрик** — выведи метрики из контекста. Если бот работает с базой на
1000 записей — это метрика. Если API обрабатывает заказы — прикинь объём.
Используй осторожные формулировки: «до 1000+», «от 50».

**NDA неочевиден** — по умолчанию ставь `isNda: false`. Ставь `true` только если
в переписке явно обсуждалась конфиденциальность или если проект для крупной
узнаваемой компании.

**Проект слишком простой** — даже простая задача (поправить скрипт, настроить сервер)
может стать кейсом. Фокусируйся на проблеме заказчика и результате, а не на объёме
кода.
