# Функциональные требования: сайт-портфолио бэкенд-разработчика

## Контекст

**Целевая аудитория:** заказчики на фрилансе
**Главная задача сайта:** продать экспертизу, вызвать доверие и конвертировать посетителя в обращение
**Специфика:** большинство проектов — бэкенд, закрытый доступ, на часть проектов действует NDA

---

## 1. Структура сайта

### 1.1. Главная страница (Landing)

**Цель:** за 10 секунд дать понять, кто ты, что делаешь и почему стоит работать с тобой.

| Блок | Описание |
|------|----------|
| Hero-секция | Краткий заголовок-оффер (что ты делаешь + для кого), подзаголовок с ключевой экспертизой, CTA-кнопка «Обсудить проект» |
| Цифры / факты | 3–5 метрик доверия: количество проектов, лет опыта, ключевые технологии, отрасли |
| Превью проектов | Карточки 3–4 лучших кейсов со ссылкой на полный список |
| Краткий блок «Обо мне» | 2–3 предложения + ссылка на полную страницу |
| CTA-блок | Повторный призыв к действию внизу страницы |

---

### 1.2. Проекты (Портфолио)

**Цель:** убедить заказчика, что ты решаешь задачи, а не просто пишешь код.

#### Список проектов

- Сетка карточек (7+ проектов)
- **Фильтрация** по категориям: тип проекта (бот, API, сервис, интеграция и т.д.), технологии / стек, отрасль заказчика
- Каждая карточка содержит: название (или обезличенное название для NDA-проектов), краткое описание в 1–2 строки, теги технологий, иконка/превью

#### Страница отдельного проекта (кейс)

Структура каждого кейса строится по формуле **«Задача → Решение → Результат»**:

| Секция | Содержание |
|--------|------------|
| Заголовок | Название проекта или обезличенное описание («Бот для автоматизации HR-процессов в логистической компании») |
| Задача | Что нужно было сделать, какая проблема была у заказчика |
| Решение | Что именно ты сделал, какие технические решения принял и почему |
| Схема архитектуры | Визуальная диаграмма: компоненты системы, взаимодействие сервисов, используемые технологии |
| Техстек | Список технологий с иконками (языки, фреймворки, БД, инфраструктура) |
| Скриншоты / видео | Демонстрация интерфейсов (если есть), записи экрана работы бота/сервиса, замазанные/обезличенные скриншоты для NDA-проектов |
| Результат | Конкретные метрики: что улучшилось, сколько сэкономлено, какие показатели выросли |
| Моя роль | Что конкретно делал ты (если работал в команде) |

#### Работа с NDA-проектами

Для проектов под NDA предусмотреть особый подход:

- Обезличенное название: отрасль + тип задачи вместо имени компании (например, «Платёжный шлюз для маркетплейса»)
- Размытые / схематичные скриншоты или замена на мокапы
- Архитектурные схемы без упоминания конкретного заказчика
- Пометка «NDA» в виде бейджа на карточке — это нормально и вызывает доверие, показывая, что ты ответственно относишься к конфиденциальности

---

### 1.3. Обо мне

**Цель:** дать заказчику понять, с кем он будет работать.

| Блок | Описание |
|------|----------|
| Краткая биография | Кто ты, сколько лет в разработке, на чём специализируешься |
| Навыки и технологии | Визуальная карта стека: языки, фреймворки, БД, инструменты. Сгруппировать по категориям (бэкенд, базы данных, DevOps, интеграции и т.д.) |
| Опыт работы | Таймлайн или список ключевых мест работы / крупных проектов |
| Подход к работе | 3–4 пункта: как выстраиваешь процесс, коммуникацию, сроки. Заказчикам важно понимать не только ЧТО ты делаешь, но и КАК |
| Скачать резюме (опционально) | Кнопка для скачивания PDF-резюме |

---

### 1.4. Контакты

**Цель:** максимально упростить первый шаг к обращению.

| Элемент | Описание |
|---------|----------|
| Форма обратной связи | Поля: имя, email/Telegram, описание задачи. Минимум полей — максимум конверсии |
| Прямые контакты | Telegram, email, ссылки на профили (GitHub, LinkedIn, фриланс-площадки) |
| CTA-текст | Мотивирующий заголовок перед формой: «Есть задача? Давайте обсудим» |

## 2. Сквозные элементы

### Шапка (Header)

- Логотип / имя
- Навигация: Проекты, Обо мне, Контакты
- CTA-кнопка «Обсудить проект» (всегда видна)

### Подвал (Footer)

- Контакты (Telegram, email)
- Ссылки на соцсети / профессиональные профили
- Копирайт

## 3. Рекомендации по подаче бэкенд-проектов

Поскольку бэкенд сложно «показать», для каждого проекта стоит подготовить:

1. **Архитектурная диаграмма** — визуально показывает масштаб и сложность решения. Можно рисовать в draw.io, Mermaid или Figma. Это самый мощный инструмент для бэкенд-портфолио.

2. **Скринкаст / GIF работы** — даже если пользователь не может зайти сам, 30-секундное видео работы бота или интерфейса показывает, что проект живой и рабочий.

3. **Метрики и цифры** — «Обрабатывает 10 000 запросов в день», «Сократил время обработки заявок с 2 часов до 5 минут». Цифры убеждают лучше любого описания.

4. **Фрагменты кода (по желанию)** — небольшие примеры элегантных решений, паттернов или подходов. Показывают качество кода без раскрытия всего проекта.

## 4. Приоритизация функционала

### MVP (первый запуск)

- Главная страница с hero, цифрами и превью проектов
- Страница списка проектов с карточками и фильтрацией по тегам
- Детальная страница кейса (задача → решение → результат)
- Страница «Обо мне»
- Форма обратной связи + прямые контакты
- Адаптивная вёрстка (мобильная версия обязательна)

### Второй этап (после запуска)

- SEO-оптимизация мета-тегов и структуры
- Анимации и микровзаимодействия
- Интеграция формы с Telegram-ботом (уведомления о заявках)
- Аналитика (Яндекс.Метрика / Google Analytics)
- Open Graph разметка для красивых превью при шаринге ссылки

---

## Резюме

Ключевой принцип сайта — **«Покажи результат, а не код»**. Заказчику на фрилансе не важно, как красиво написан код. Ему важно, что ты понимаешь задачи бизнеса, умеешь их решать и можешь это доказать. Архитектурные схемы, метрики, структурированные кейсы и грамотная работа с NDA-проектами — именно это отличает сильное бэкенд-портфолио от списка технологий.
